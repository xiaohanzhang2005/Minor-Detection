# 未成年人识别 Skill 工作推进计划（详细版）

## 1. 总体目标

本项目后续工作分为三条并行但职责清晰的主线：

1. 构建一个可交付、可复用、符合开放规范的未成年人识别 Skill。
2. 构建一个调用该 Skill 的外部运行时系统与体验网站。
3. 保留并升级当前自迭代研发链路，使其服务于正式 Skill 的持续优化。

最终交付不再等同于“当前冠军版 prompt”，而是一个由以下部分组成的完整体系：

- 正式 Skill 包
- 外部运行时系统
- 网站体验系统
- 自迭代优化与评测工具链
- 配套文档与接入规范

---

## 2. 产品与交付边界

### 2.1 正式 Skill 的定位

正式 Skill 是一个面向 Agent/系统调用的能力包，其核心职责是：

- 接收当前对话、可选历史会话、可选结构化上下文
- 基于 ICBO 框架完成未成年人识别
- 输出统一结构化 JSON
- 产出用户画像、证据归因、风险等级、趋势分析

正式 Skill 不负责：

- 持久化数据库管理
- 长期用户画像自动存储
- 网站界面展示
- 在线服务部署
- 自迭代训练流程

### 2.2 外部运行时系统的定位

运行时系统负责：

- 组织输入数据
- 调用 Skill 的统一入口，由 Skill 内部按条件决定是否调用时间脚本
- 在需要时为 Skill 提供外置 RAG 结果或相关配置
- 查询历史画像或历史会话
- 构造传给 Skill 的标准化 payload
- 保存分析结果
- 为网站/API/后台提供统一调用接口

### 2.3 自迭代研发链路的定位

研发链路负责：

- 评估正式 Skill
- 发现边界错误
- 优化 Skill 正文中的推理规则
- 优化 Skill 的触发 description
- 管理版本演化和回归验证

## 3. Skill 的完整设计

### 3.1 Skill 的核心能力

正式 Skill 应内建以下核心能力：

1. 未成年人概率判断
2. 风险等级识别
3. ICBO 特征提取
4. 用户建模
5. 跨会话趋势分析
6. 证据分类与冲突解释
7. 可选 RAG 证据消费
8. 可选时间特征消费

### 3.2 Skill 的触发场景

Skill 的 `description` 应覆盖这些场景：

- 判断对话者是否可能是未成年人、青少年、中学生、高中生
- 对聊天记录做年龄倾向、学生倾向、未成年人风险分析
- 基于多轮聊天或历史会话进行用户画像分析
- 输出未成年人概率、证据、置信度解释、趋势变化
- 识别校园、学业、家长、同伴、青少年表达风格等线索
- 即使用户未明确说“未成年人识别”，但本质需求是判断“像不像未成年用户”时也应触发

### 3.3 Skill 的输入模式

正式 Skill 统一使用 `analysis_payload` 思想，支持三种输入模式。

#### 模式 A：单会话模式

适用于：

- 单次对话
- 一个用户的一个 session

输入内容：

- 当前会话对话记录

输出重点：

- 当前未成年人概率
- 当前会话用户画像
- 当前证据说明

#### 模式 B：多会话历史模式

适用于：

- 同一用户跨对话跨时间聊天记录
- 需要建模用户趋势

输入内容：

- 多个 session
- 每个 session 的时间信息

输出重点：

- 聚合后的用户画像
- 每段历史的概率变化轨迹
- 当前综合概率
- 趋势总结

#### 模式 C：外部增强摘要模式

适用于：

- 外部系统已维护长期画像
- 不希望每次都传全量历史

输入内容：

- 当前会话
- 外部系统整理好的画像摘要
- 可选 RAG 摘要
- 可选时间特征

输出重点：

- 结合摘要后的综合判定
- 冲突证据解释
- 当前结论与历史摘要关系说明

### 3.4 Skill 的标准输入结构

建议统一为如下概念结构：

```json
{
  "mode": "single_session | multi_session | enriched",
  "conversation": [],
  "sessions": [],
  "context": {
    "time_features": {},
    "retrieved_cases": [],
    "prior_profile": {},
    "channel": "",
    "locale": ""
  }
}
```

字段要求：

- `conversation` 用于单会话模式
- `sessions` 用于多会话模式
- `context.time_features` 用于结构化时间特征；可来自输入原文解析或时间脚本规范化
- `context.retrieved_cases` 用于 RAG 结果
- `context.prior_profile` 用于外部运行时摘要注入

### 3.5 Skill 的标准输出结构

正式 Skill 默认输出面向系统消费，而不是直接为终端用户设计。

建议输出 schema：

```json
{
  "decision": {
    "is_minor": true,
    "minor_confidence": 0.82,
    "confidence_band": "high",
    "risk_level": "Low"
  },
  "user_profile": {
    "age_range": "14-16岁",
    "education_stage": "初中/高中",
    "identity_markers": ["学生", "考试压力"]
  },
  "icbo_features": {
    "intention": "",
    "cognition": "",
    "behavior_style": "",
    "opportunity_time": ""
  },
  "evidence": {
    "direct_evidence": [],
    "historical_evidence": [],
    "retrieval_evidence": [],
    "time_evidence": [],
    "conflicting_signals": []
  },
  "reasoning_summary": "",
  "trend": {
    "trajectory": [],
    "trend_summary": ""
  },
  "uncertainty_notes": [],
  "recommended_next_step": ""
}
```

### 3.6 Skill 内部的用户建模设计

用户建模明确属于 Skill 内部核心能力。

Skill 内部负责：

- 从当前会话中提取用户画像
- 从多会话历史中归纳稳定画像
- 生成教育阶段、年龄段、身份标记
- 在有历史输入时计算概率变化轨迹
- 生成趋势总结

Skill 内部不负责：

- 将画像长期写入数据库
- 为后续会话自动维护持久化记忆

这里要明确区分两种概念：

- 推断型用户建模：属于 Skill 内
- 持久化型长期画像：属于 Skill 外

### 3.7 Skill 中证据优先级规则

应明确规定以下优先级：

1. 当前对话中的直接证据
2. 多会话历史中的稳定证据
3. RAG 检索辅助证据
4. 时间特征辅助证据

裁决原则：

- 直接证据优先于辅助证据
- 历史证据可增强或削弱当前结论，但不能无条件覆盖直接证据
- RAG 与时间特征均为弱证据
- 证据冲突时优先降低置信度，不轻易翻转标签

### 3.8 Skill 内部的 RAG 设计

正式 Skill 应支持可选内置 RAG，但不强依赖 RAG 才能运行。

RAG 设计原则：

- `无 RAG` 时，Skill 也可独立完成基础识别
- `内置 RAG` 时，Skill 内部工作流可调用本地检索脚本
- `外置 RAG` 时，运行时系统可将外部检索结果标准化后注入，Skill 只消费 `retrieved_cases`

资产来源规则：

- 内置 RAG 的运行资产可以由 benchmark 训练集导出生成
- 但进入 Skill 的应是专门打包的 retrieval assets，而不是原始 `train.jsonl`
- `val/test/trigger eval` 等研发评测资产不得进入 Skill

Skill 不应关心检索结果来自哪里，只关心：

- 结果格式是否标准化
- 如何使用这些结果
- 与直接证据冲突时如何处理

### 3.9 Skill 内部的时间脚本设计

时间脚本应明确允许内置在 Skill 中，但形式应为脚本而不是冗长 prompt 规则。

建议内置：

- `scripts/extract_time_features.py`

输入：

- 时间戳
- 时区
- 可选地区

输出：

- 星期几
- 是否深夜
- 是否周末
- 时间桶
- 可选假期标签

Skill 使用原则：

- 当前阶段优先直接使用输入中显式出现的时间描述作为时间证据
- 只有当输入时间信息不完整、格式不统一，或缺少稳定派生字段时，才启用时间脚本
- 时间特征只能作为辅助证据
- 不得仅凭深夜或周末直接判定为未成年人

补充说明：

- “Skill 内置时间脚本”和“运行时系统调用时间脚本”不是两套逻辑
- 推荐做法是：时间脚本作为 Skill 包的一部分交付，并由 Skill 内部工作流按条件调用
- 如果输入里已经有像“2026-03-18 周三 8:59”这样的完整时间描述，则主判定可直接使用，不必强制跑脚本
- `SKILL.md` 需要明确写清时间脚本的适用条件，例如“仅在日期缺少星期、假期标签或稳定时间桶时调用脚本”

### 3.10 Skill 的目录结构建议

```text
minor-detection/
├── SKILL.md
├── references/
│   ├── evidence-rules.md
│   ├── icbo-guidelines.md
│   └── output-schema.md
├── scripts/
│   ├── extract_time_features.py
│   └── retrieve_cases.py
└── assets/
    └── retrieval_assets/
        ├── corpus.jsonl
        ├── index.pkl
        └── manifest.json
```

## 4. 外部运行时系统设计

### 4.1 运行时系统职责

外部运行时系统负责把外部世界的信息整理成 Skill 能消费的标准输入。

它应承担：

- 原始输入标准化
- 时间特征提取
- RAG 结果构造
- 历史画像摘要构造
- Skill 调用
- 分析结果存档
- 网站/API/后台输出渲染

### 4.2 运行时系统模块划分

建议拆分为：

#### `input_normalizer`

负责：

- 接收原始对话
- 接收 JSON/JSONL 上传
- 接收多会话历史
- 统一转成 Skill 标准输入结构

#### `time_feature_adapter`（可选高级集成）

负责：

- 在某些宿主系统希望显式预填时间特征时，复用 Skill 包内同一份时间脚本
- 生成 `time_features`

说明：

- 默认交互路径不要求宿主系统直接调用该模块
- 对外默认仍是“只调用 Skill 统一入口”

#### `external_rag_adapter`（可选高级集成）

负责：

- 在宿主系统已有外置检索能力时，对接外部 RAG 服务
- 将外部检索结果标准化为 `retrieved_cases`

说明：

- 内置 RAG 仍由 Skill 内部工作流按条件调用
- 默认交互路径不要求宿主系统显式使用该模块

#### `profile_builder`

负责：

- 从多会话历史中提炼摘要
- 从数据库中拉取长期画像快照
- 生成 `prior_profile`

#### `skill_executor`

负责：

- 调用正式 Skill
- 约束输出结构

#### `result_renderer`

负责：

- 把 JSON 渲染成网站展示内容
- 生成图表、趋势卡片、证据面板

#### `profile_store`

负责：

- 长期保存用户画像与历史分析结果
- 为未来分析提供摘要输入

### 4.3 长期画像系统设计

长期画像系统只放在 Skill 外部。

它负责：

- 保存每次分析结果
- 保存每次画像快照
- 保存历史概率轨迹
- 按用户查询历史
- 下次分析时只传摘要，不必重传全量历史

推荐存储字段：

- `user_id`
- `session_id`
- `session_time`
- `is_minor`
- `minor_confidence`
- `risk_level`
- `profile_snapshot`
- `evidence_summary`
- `trend_snapshot`

### 4.4 RAG 的两种部署方式

#### 本地内置 RAG

适用于：

- 演示环境
- 单机部署
- 离线环境

特点：

- Skill 包含固定索引或固定语料
- 由 Skill 内部工作流按条件调用本地检索脚本

资产要求：

- 本地内置 RAG 使用的是导出的 retrieval assets
- 不直接把 benchmark 的训练/验证/测试集按原样打包进 Skill

#### 外部 RAG 服务化

适用于：

- 正式系统
- 持续更新
- 多个系统共享同一检索服务

特点：

- 运行时系统通过 API 获取检索结果
- Skill 不直接感知服务实现

推荐优先级：

- 有外置 RAG 时优先外置
- 否则回退到内置 RAG
- 都没有时仍可运行

## 5. 网站体验系统设计

### 5.1 网站定位

网站不是简单前端，而是一个体验展示站。

它的核心目标：

- 让老板和评审能体验系统效果
- 展示 Skill 的能力与解释性
- 展示方法论、数据来源和自迭代过程
- 为未来商业化或合作方接入做展示

### 5.2 网站核心页面

#### 首页

展示：

- 产品定位
- 能力概览
- Skill 与系统关系说明
- 方法论概览

#### 在线体验页

支持：

- 输入单会话聊天记录
- 输入多会话历史
- 上传 JSON/JSONL
- 切换 RAG/时间增强/历史摘要模式

#### 结果页

展示：

- 未成年人概率
- 风险等级
- 用户画像
- ICBO 分析
- 关键证据
- 冲突证据
- 趋势曲线
- 总结性解释

#### 方法说明页

展示：

- ICBO 框架
- 用户建模逻辑
- RAG 的作用
- 时间特征的作用
- 自迭代优化流程
- 数据集概览

#### 开发者接入页

展示：

- 输入格式
- 输出 schema
- 三种输入模式
- RAG 和长期画像如何接入

### 5.3 网站与 Skill 的关系

网站不直接替代 Skill，而是通过外部运行时系统调用 Skill。

调用链：

网站输入 -> 运行时系统 -> Skill -> 结构化结果 -> 网站渲染

## 6. 研发工具链重构计划

### 6.1 当前研发状态

当前项目已有：

- 评估器
- 优化器
- pipeline
- 数据集
- 多轮 prompt 演化版本

这些能力保留，但要从“优化一个实验 prompt”转为“优化正式 Skill”。

### 6.2 研发工具链的两类评测

#### 输出质量评测

衡量：

- 未成年人判断准确率
- 用户画像合理性
- 趋势分析质量
- RAG 边界处理
- 时间特征使用质量

#### 触发质量评测

衡量：

- Skill 的 `description` 是否该触发时触发
- 不该触发时不触发
- 是否过宽或过窄

### 6.3 自迭代的优化对象

正式阶段，自迭代应只优化：

- Skill 正文中的推理规则
- 边界案例规则
- 证据优先级规则
- 输出填写规则

不直接优化：

- 数据库设计
- 网站文案
- 目录结构
- 外部运行时 orchestration

### 6.4 Description 优化

`description` 的优化必须独立于正文优化。

应建立单独的：

- 正向触发样本
- 反向触发样本
- 训练/验证拆分
- 多轮触发率测试

### 6.5 正式研发顺序

正确顺序应为：

1. 冻结 Skill 结构
2. 冻结输入输出 schema
3. 冻结评测集
4. 让自迭代优化正文规则
5. 单独优化 `description`

## 8. 里程碑设计

### M1：Skill 规范冻结

完成标志：

- Skill 交付边界确定
- 用户建模内外边界确定
