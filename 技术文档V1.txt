### **1. 总体目标与核心思想**

本项目旨在设计并实现一个先进的AI“技能”（Skill），用于**动态、持续地识别**在线交互中的青少年用户（<18岁），并对其对话内容进行**高危风险评估**。

我们的核心思想是摒弃传统的、基于静态规则或无状态模型的检测方法。我们引入了前沿的AI Agent设计理念，构建一个包含**三大核心支柱**的系统：

1. **ICBO理论框架**：作为我们分析用户行为的“认知内核”，提供深度、结构化的特征维度。
2. **长期记忆建模（借鉴MemSkill）**：使Skill能够跨越多次独立的对话，追踪并综合用户的历史行为轨迹，实现动态画像的持续更新。
3. **离线自迭代优化（借鉴skill-creator）**：建立一套自动化的“测试-评估-优化”闭环，利用我们构建的垂域数据集，让Skill的Prompt能够自我进化，持续提升其识别的精确度和召回率。

### **2. 项目组成部分 (System Architecture)**

整个系统由三个主要部分构成：**数据基座（Data Foundation）**、**在线执行体（Online Executor Skill）**、和**离线进化引擎（Offline Evolution Engine）**。

**2.1 数据基座 (Data Foundation)**

- **构成**：一个高质量、带精细标注的**垂域Benchmark数据集**。
- **内容**：
    - **正样本**：大量真实的、符合ICBO框架的青少年对话（包含社交和知识问答场景）。
    - **负样本**：大量“高迷惑性”的成年人对话（如大学生、年轻职场人），用于训练模型的区分边界（社交和知识问答）。
- **作用与划分**：数据集将被严格、一次性地划分为三份，且永不交叉：
    - **训练集 (Train Set, ~70%)**: 用于**在线校准**（作为RAG检索库）和离线自进化时**优化师Agent的学习材料**。
    
    > **角色 A：作为“在线校准”的 RAG 检索库 (The Reference Library)**
    > 
    > - **场景**：系统已经上线，正在实时处理一个真实的、未知用户的对话。
    > - **动作**：
    >     1. 用户发来一句：“烦死了，不想上学”。
    >     2. 我们的系统立刻去**“检索库”**里搜相似的话。
    >     3. **注意！这个“检索库”里装的，仅仅是你的 Train Set。** 为什么？因为那是我们“已知且确认”的经验。
    >     4. 系统搜到了 Train Set 里的一条记录：“不想上学，作业太多（标签：初二，14岁）”。
    >     5. 系统将这条信息作为辅助提示词喂给大模型：“用户说了这句话，顺便提一下，在我们已知的经验库里，相似的话通常是14岁初中生说的，请你综合判断。”
    > - **理解**：在这里，Train Set 就像是一本**“错题本/案例集”**，在线系统在做判断时，可以随时翻开这本案例集找参考。
    > 
    > **角色 B：作为“离线自进化”时 Optimizer Agent 的学习材料 (The Learning Material)**
    > 
    > - **场景**：半夜，系统没流量了，我们开启了“离线自迭代优化”流水线。
    > - **动作**：
    >     1. Evaluator 用验证集（Val Set）考了一遍当前的 Prompt，发现它总是把“考研压力大”误判为青少年。
    >     2. 现在轮到 Optimizer（优化师 Agent）出场了。它的任务是修改 Prompt。但它怎么知道该怎么改呢？
    >     3. Optimizer 会去 **Train Set** 里寻找灵感。
    >     4. 它在 Train Set 里搜出几条真正的“高中生压力大”的例子（正样本），和几条真正的“大学生考研压力大”的例子（负样本）。
    >     5. Optimizer 通过对比这两组真实的样本，**自动总结出规律**（例如：“高中生更关注家长态度，大学生更关注导师和未来”）。
    >     6. 基于这个总结出的规律，Optimizer 写出了新版 Prompt：“在判断学业压力时，请特别注意区分关注点是在家长还是在导师/未来职业上……”
    > - **理解**：在这里，Train Set 是提供给 Optimizer 进行**“归纳推理 (Inductive Reasoning)”**的原材料。Optimizer 看着这些正确的数据，自己悟出规律，然后写进说明书（Prompt）里。
    - **验证集 (Validation Set, ~10%)**: 用于在**离线自进化循环**中，评估每个新版Prompt的性能（F1-Score），并决定是否采纳。
    - **测试集 (Test Set, ~20%)**: **绝对隔离的“最终考卷”**。只在项目完成、论文撰写前使用一次，用于报告最终的、可信的学术指标。

**2.2 在线执行体 (Online Executor Skill)**

- **构成**：这是系统上线后，实际处理用户请求的模块。它由Skill的**核心逻辑（Prompt）**、**记忆模块**和**校准模块**组成。
- **工作流**：
    1. **输入**：`user_id` 和 `current_dialogue`。
    2. **记忆检索**：根据 `user_id`，从长期记忆库（如本地向量数据库）中检索该用户的历史ICBO画像。
    3. **上下文合成**：将“历史画像”与“当前对话”整合成一个丰富的上下文。
    4. **RAG校准**：提取当前对话的关键特征，在**训练集（Train Set）**中检索最相似的K个样本，作为“参考案例”补充进上下文。
    5. **LLM推理**：将合成后的完整上下文，喂给由`skill.md`定义的Prompt，调用通用大模型（如**Gemini 3 Flash**）进行分析。
    6. **输出与记忆更新**：输出当前的风险判断，并将新生成的ICBO特征更新回该用户的长期记忆库。

**2.3 离线进化引擎 (Offline Evolution Engine)**

- **构成**：这是在研发环境中运行的自动化流水线，由**评估器（Evaluator）**和**优化器（Optimizer）**两个核心Agent组成。
- **工作流**：
    1. **评估**：`Evaluator` Agent加载**验证集（Val Set）**，逐条测试当前版本的“在线执行体Skill”（**GPT-4o**），并计算出整体的F1-Score、Precision、Recall。
    2. **诊断**：`Evaluator` Agent生成一份详细的**错误分析报告**，列出所有判断错误的案例（特别是False Positives和False Negatives）。
    3. **自动优化**：`Optimizer` Agent读取当前的`skill.md`和错误分析报告，调用强大的LLM（如**Claude 3.5/4 Sonnet ？**），让其**自动重写`skill.md`**，旨在修复报告中指出的缺陷。
    4. **迭代**：系统用新生成的`skill_v_next.md`替换旧版，再次启动评估流程。如果新版在验证集上的F1-Score更高，则保留；否则回滚。此过程循环进行，直到性能收敛。

### **3. 关于参数与微调的辨析**

- **之前设计的参数是什么？**
    - 是在`user_profile.py`里**手动写死**的规则权重，例如`self.minor_probability += 0.2`。
- **我们现在的做法是什么？**
    - **不再手动调这些参数了！这些“魔法数字”应该被移除**，或者作为**可进化对象**。
- **训练集用来微调参数？**
    - 不完全是。在这个框架下，`训练集`的主要作用是作为**在线校准的参考库**和**离线优化Prompt的灵感来源**。
    - 我们**不微调模型参数（Fine-tuning the model's weights）**，我们**“微调”Prompt（Fine-tuning the prompt）**。这是一个更轻量、更灵活的范式，也是`skill-creator`的核心思想。
- **验证集用来做什么？**
    - **它的唯一作用，就是在离线进化引擎中，充当“模拟考卷”，为每一轮Prompt的迭代提供客观的分数反馈。**
- **测试集用来做什么？**
    - **它的唯一作用，就是在整个研发过程结束后，提供一次最终的、公正的“高考成绩”，写进论文里。**
