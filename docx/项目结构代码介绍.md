# 项目结构代码介绍

1. 现在真正跑起来的主链是什么。
2. 各目录各文件在主链里的位置是什么。
3. 哪些是历史兼容代码，哪些是临时残留，哪些可以作为删除候选。

## 1. 项目现在到底在干什么

项目的当前重心不是“聊天演示站”，而是“围绕 bundled skill 做可迭代优化闭环”。

你可以把它理解成：

- 输入是一批分层采样的 benchmark 样本。
- 中间执行层有两种：Mode A 让 Agent 去调 skill；Mode B 直接调 skill。
- skill 内部再按自己的总控脚本做时间处理、检索(RAG)、分类、schema 修复。
- 输出不是只看单样本结果，而是看 judge 聚合后的 report、failure packets、protected packets。
- optimizer 再根据 judge 结果去改 skill，compare 决定 promote 还是 rollback。
- 最后进入人审和 final test。

## 2. 两条主链

### 2.1 Mode A

```text
scripts/run_skill_iteration_loop.py
  -> src/skill_loop/loop.py
  -> src/skill_loop/runner.py
  -> Agent 执行预制 launcher
  -> skills/minor-detection/scripts/run_minor_detection_pipeline.py
  -> src/skill_loop/judge.py
  -> src/evolution/optimizer.py
  -> src/skill_loop/compare.py
  -> 人审
```

关键点：

- `runner.py` 会把 skill 安装到隔离 `codex_home`，避免污染真实技能环境。
- Agent 被限制成“只运行一次预制 launcher，不自己乱改文件”。
- 每个样本都会留下 transcript、tool_trace、observability、agent_output、timing、run_metadata。
- judge 是基于 artifacts 判定，而不是只基于最终 JSON。

### 2.2 Mode B

```text
scripts/run_direct_iteration_loop.py
  -> src/skill_loop/loop.py
  -> src/skill_loop/direct_runner.py
  -> skills/minor-detection/scripts/run_minor_detection_pipeline.py
  -> src/skill_loop/judge.py
  -> src/evolution/optimizer.py
  -> src/skill_loop/compare.py
  -> 人审
```

关键点：

- `direct_runner.py` 不经过 Agent，直接跑 skill 总控脚本。
- 除了执行层不同，其余 judge、optimizer、compare、versioning 都和 Mode A 共用。

## 3. 目录级别怎么看

### 3.1 根目录

- `README.md`
  - 项目入口说明，适合先快速看。

- `app_formal.py`
  - 一个正式 runtime 的演示页，方便人工检查 `conversation -> context -> output`。
  - 不属于 A/B 主链，但对调试 formal runtime 有帮助。

- `app_v2.py`
  - 旧版 Web 演示页，走 executor + external RAG + memory 体系。
  - 这是历史兼容代码，不是当前主链。

- `requirements.txt`
  - 依赖列表。

### 3.2 `src/`

这是“项目级 Python 代码”。里面既有当前主链，也有旧兼容层。

#### `src/config.py`

- 统一定义路径：`ROOT_DIR`、`DATA_DIR`、`SKILLS_DIR`、benchmark split 路径等。
- 维护 `skills/active_version.txt` 指向的 active skill。
- 老 runtime 和新 loop 都会用到它。

#### `src/models.py`

- 项目的数据合同中心。
- 定义了：
  - `AnalysisPayload`
  - `FormalSkillOutput`
  - `SkillOutput`
  - 各类 formal 子结构
- judge、runtime、skill 输出修复都依赖这些模型。

#### `src/utils/path_utils.py`

- 把绝对路径转成项目内相对路径。
- 用于 run report、loop summary、judge report，让产物可迁移、可读。

#### `src/utils/llm_client.py`

- 老执行链和 optimizer 共用的 OpenAI-compatible client。
- 带 retry、structured output repair、metrics 统计。

### 3.3 `src/skill_loop/`

这是当前主链最重要的一组代码。

- `__init__.py`
  - 暴露 loop 框架公共入口。

- `loop.py`
  - 完整闭环总控。
  - 负责：baseline snapshot、evaluate、judge、optimize、compare、promote、总结输出。

- `runner.py`
  - Mode A 的执行器。
  - 负责：
    - 分层采样
    - 建立隔离 `codex_home`
    - 安装 skill 快照
    - 为每个样本写 `payload.json` 和 `run_skill_once.py`
    - 调 `codex exec`
    - 收集 transcript / stderr / tool trace / observability / metadata / manifest

- `direct_runner.py`
  - Mode B 的执行器。
  - 结构基本和 `runner.py` 对齐，但直接调用 skill 总控脚本。

- `judge.py`
  - 把 sample 级 artifacts 变成 judge 级别结论。
  - 会产出：
    - `judge_report.json`
    - `error_index.jsonl`
    - `protected_index.jsonl`
    - `failure_packets/`
    - `protected_packets/`

- `compare.py`
  - 判断 candidate 能不能 promote。
  - 当前门禁包括：
    - F1 要提升
    - schema validity 不退化
    - invocation success 不退化
    - step compliance 不退化
    - protected 样本不能回归

- `versioning.py`
  - skill 版本命名、快照、库存、清理预览。
  - 对 `minor-detection-v0.1.0`、`rc` 候选版本、带 run tag 的稳定版本都在这里管理。

- `schema_consistency.py`
  - 防止 prompt/markdown 合同和 Python formal schema 漂移。
  - loop 在跑数据前会先检查它。

- `packaging.py`
  - 调 `claude-skill-creator` 做 skill 校验和 `.skill` 打包。

- `validation_seed.py`
  - 生成 Mode A 自迭代验收用的“临时坏 baseline”。
  - 通过在 markdown 规则层面故意降级，稳定制造可优化错误。

### 3.4 `src/evolution/`

- `optimizer.py`
  - 当前主链里真正还在用的是它。
  - 它根据 judge failure / protected packets 去改 bundled skill 的文本和配置。

- `evaluator.py`
  - 更偏旧链路的 evaluator。
  - 还可做历史验收或兼容检查，但已不是当前 A/B 主链的核心。

### 3.5 `skills/`

这里是“skill 目录树”，不是普通代码目录，而是项目主产物的一部分。

#### `skills/minor-detection/`

当前 source-of-truth bundled skill。

- `SKILL.md`
  - skill 本体说明书。
  - optimizer 的很多改动会最终反映到这里和它引用的 references 里。

- `references/`
  - 规则文档、output schema、classifier prompt 模板、retrieval query 模板等。

- `scripts/run_minor_detection_pipeline.py`
  - skill 内部总控脚本。
  - 这是 A/B 主链真正调用的核心文件。

- `scripts/extract_time_features.py`
  - 时间脚本入口。

- `scripts/retrieve_cases.py`
  - 内置检索脚本入口。

- `scripts/config.py`
  - skill 内部运行时配置。

- `scripts/_payload_normalizer.py`
  - 把外部 payload 统一转成 skill 内部可处理结构。

- `scripts/_classifier_client.py`
  - 分类器请求封装。

- `scripts/_schema_repair.py`
  - 输出校验与修复。

- `scripts/_profile_merge.py`
  - 把 classifier 输出补齐成稳定 formal 输出形状。

- `scripts/_skill_retrieval_utils.py`
  - skill 自己的 retrieval text builder 和 retriever。

- `scripts/_skill_time_utils.py`
  - skill 自己的时间工具。

#### 其他带版本号的 skill 目录

- `minor-detection-v0.1.0`
  - 当前 loop 默认 baseline snapshot。

- `minor-detection-v0.1.1-rc...`
  - 过去某次 run 生成的 candidate 目录。

- `minor-detection-verifya-...`
  - Mode A validation seed 用的临时坏 baseline。

### 3.6 `scripts/`

这里是项目级 CLI 入口和维护工具。

#### 当前主链入口

- `run_skill_iteration_loop.py`
  - Mode A 入口。

- `run_direct_iteration_loop.py`
  - Mode B 入口。

- `run_final_test.py`
  - final test 入口，可切换 `agent` 或 `direct` runner。

#### 当前保留的维护工具

- `list_skill_versions.py`
  - 看 skill 版本库存和 cleanup preview。

- `cleanup_skill_versions.py`
  - 删旧 snapshot 的安全入口。

- `prepare_mode_a_validation_seed.py`
  - 生成 Mode A 自迭代验收用坏 baseline 的 CLI 包装。

- `inspect_formal_sample.py`
  - 单样本调试工具，适合排查 payload enrich、retrieval、formal output。

- `export_retrieval_assets.py`
  - 把 retrieval index 复制到 skill 目录中。

- `prepare_data.py`
  - 数据集准备工具，偏上游支撑，不属于当前讲解主链重点。

#### 历史兼容脚本

- `run_pipeline.py`
  - 旧版离线总控。
  - 现在已经不是主链。

- `run_acceptance_suite.py`
  - 旧版 acceptance suite。
  - 现在已经不是主链。

### 3.7 `test/`

- `test_skill_loop.py`
  - 当前最重要测试文件。
  - 覆盖 runner、direct_runner、judge、compare、versioning、loop 等主链能力。

- `test_validation_seed.py`
  - 覆盖 validation seed 逻辑。

- `test_versioning.py`
  - 覆盖 skill snapshot / inventory / cleanup 逻辑。

- `test_formal_skill_runtime.py`
  - 覆盖 formal runtime adapter 和 bundled skill helper 行为。

- `test_mvp_demo.py`
  - 旧 executor demo 测试。

- `test_rag_demo.py`
  - 旧 external RAG demo 测试。

- `test_technical_doc_v1.py`
  - 很大的历史集成测试文件，更多是旧路线沉淀。

### 3.8 `tmp/` 与 `reports/`

这两个目录都不应该再被当成“主项目代码目录”看。

- `reports/`
  - 是运行产物仓库。
  - 里面即使出现 `.py`，本质也是历史副本或临时 launcher，不是维护源代码。

- `tmp/`
  - 是草稿、试验、历史 skill 副本、validation seed 中转区。
  - 里面的 `.py` 大多不是应继续维护的正式代码。

## 4. 删除候选梳理

下面这部分是“删除也不影响当前 A/B 主链”的候选清单，但我建议按风险分层处理，而不是一口气全删。

### 4.1 第一优先级：明显不影响主链，可优先清理

- `tmp/*.py`
  - 用途：最早期原型、旧 calibrator、临时测试脚本。
  - 是否影响主链：不影响。
  - 建议：删。

- `tmp/history/`
  - 用途：旧 prompt/skill 草稿历史。
  - 是否影响主链：不影响。
  - 建议：删。

- `tmp/iterations/`
  - 用途：早期迭代产物快照。
  - 是否影响主链：不影响。
  - 建议：删。

- `tmp/skill_validation/`
  - 用途：旧 validation seed 残留目录。
  - 是否影响主链：不影响。
  - 建议：删。

- `__pycache__/`、`.pytest_cache/`、`.tmp_tests/`
  - 用途：缓存与临时测试目录。
  - 是否影响主链：不影响。
  - 建议：删。

### 4.2 第二优先级：不影响主链，但属于“旧链路代码”，删前最好确认没人还在手动用

- `app_v2.py`
  - 用途：旧网页演示。
  - 是否影响主链：不影响。
  - 建议：大概率可删。

- `scripts/run_pipeline.py`
  - 用途：旧总控脚本。
  - 是否影响主链：不影响。
  - 建议：大概率可删。

- `scripts/run_acceptance_suite.py`
  - 用途：旧 acceptance 脚本。
  - 是否影响主链：不影响。
  - 建议：大概率可删。

- `test/test_mvp_demo.py`
  - 用途：旧 demo smoke test。
  - 是否影响主链：不影响。
  - 建议：可删。

- `test/test_rag_demo.py`
  - 用途：旧 external RAG smoke test。
  - 是否影响主链：不影响。
  - 建议：可删。

- `test/test_technical_doc_v1.py`
  - 用途：旧路线的大型集成测试。
  - 是否影响主链：不影响。
  - 建议：可删，且删它能明显减轻维护负担。

### 4.3 第三优先级：虽然不在主链，但建议先保留

- `app_formal.py`
  - 用途：formal runtime 演示。
  - 是否影响主链：不影响主链，但方便人工看单样本行为。
  - 建议：先保留。

- `src/runtime/*`
  - 用途：formal runtime 适配层。
  - 是否影响主链：不影响主链，但会影响 demo 和 inspection 工具。
  - 建议：先保留。

- `src/executor/*`
  - 用途：旧 executor 兼容层。
  - 是否影响主链：不影响当前 A/B loop，但会影响旧脚本和部分工具。
  - 建议：先保留，等旧脚本一并下线再删。

- `src/retriever/*`、`src/memory/*`
  - 用途：旧 external RAG / memory 兼容层。
  - 是否影响主链：不影响 A/B loop，但影响旧演示和旧验收链。
  - 建议：先保留。

- `claude-skill-creator/`
  - 用途：skill 校验与打包依赖。
  - 是否影响主链：会影响 `src/skill_loop/packaging.py`。
  - 建议：不能删。

## 5. 如果之后要继续瘦身，建议的顺序

1. 先删 `tmp/` 下正式确认无用的 Python 和历史目录。
2. 再删缓存目录和过旧 `reports/` 运行产物。
3. 再处理旧链路代码：`app_v2.py`、`scripts/run_pipeline.py`、`scripts/run_acceptance_suite.py`、旧 demo tests。
4. 最后再考虑是否下线 `src/executor/`、`src/retriever/`、`src/memory/`、`src/runtime/` 这整套兼容层。

## 6. 建议接手阅读顺序

如果要快速理解项目，建议按下面顺序读：

1. `README.md`
2. `scripts/run_skill_iteration_loop.py`
3. `src/skill_loop/loop.py`
4. `src/skill_loop/runner.py`
5. `src/skill_loop/judge.py`
6. `src/evolution/optimizer.py`
7. `src/skill_loop/compare.py`
